# 분산 환경과 서비스

> 어디에 내놔도 당당한 내 어플리케이션

<br>

1. 랩스도 다양한 사업적인 기회들을 접하고 있다
2. 내 로컬에서는 돌아가는데요? ...가 아니기 위해서
3. 인터넷: 글로벌 컴퓨터 네트워크로, 컴퓨터 간에 상호 연결된 시스템
4. 웹 서비스: 네트워크 상에서 서로 다른 종류의 컴퓨터들 간에 상호작용을 하기 위한 소프트웨어 시스템

<br>

분산 환경에서 웹 어플리케이션을 개발할 때 고려해야 할 몇 가지 점들이 있습니다. 아래는 분산 환경에서 웹 어플리케이션을 구축할 때 고려해야 할 주요 사항입니다:

1. **성능과 확장성:**
    - 분산 환경에서는 서버의 부하 분산과 확장성을 고려해야 합니다.
    - 로드 밸런서를 사용하여 트래픽을 여러 서버로 분산시키고, 필요한 경우 서버를 추가로 확장할 수 있도록 설계해야 합니다.
2. **데이터 일관성과 동기화:**
    - 여러 서버 간의 데이터 일관성을 유지해야 합니다.
    - 데이터베이스 동기화, 캐싱 전략, 분산 트랜잭션 등을 고려하여 데이터 일관성을 보장해야 합니다.
3. **보안과 인증:**
    - 분산 환경에서는 보안을 강화해야 합니다.
    - 토큰 기반 인증, HTTPS, API 보안 등을 적용하여 데이터 유출과 인증 문제를 방지해야 합니다.
4. **네트워크 지연과 대역폭:**
    - 서버 간의 통신에는 네트워크 지연과 대역폭을 고려해야 합니다.
    - CDN (콘텐츠 전송 네트워크)를 활용하여 정적 리소스를 빠르게 제공하고, 최적화된 프로토콜을 선택해야 합니다.
5. **분산 시스템 아키텍처:**
    - 마이크로서비스, 컨테이너, 서버리스 등의 분산 시스템 아키텍처를 고려해야 합니다.
    - 각 서비스 간의 상호작용, 데이터 흐름, 확장성을 고려하여 설계해야 합니다.
6. **모니터링과 로깅:**
    - 분산 환경에서는 모니터링과 로깅을 철저히 수행해야 합니다.
    - 서버 상태, 트래픽, 오류 로그 등을 모니터링하여 문제를 빠르게 파악하고 대응해야 합니다.
7. **클라우드 서비스 활용:**
    - 클라우드 서비스를 활용하여 서버 관리를 간소화하고 확장성을 높일 수 있습니다.
    - AWS, Azure, Google Cloud 등의 클라우드 서비스를 적극 활용해보세요.

이러한 고려 사항을 바탕으로 분산 환경에서 웹 어플리케이션을 개발하면 안정성과 성능을 향상시킬 수 있습니다.

<br>

\*\*서비스 지향 아키텍처 (SOA)\*\*는 비즈니스 애플리케이션을 생성하는 소프트웨어 개발 방식입니다. 이를 통해 서로 다른 시스템 내의 서비스를 재사용하거나 독립적인 여러 서비스를 결합하여 복잡한 태스크를 수행할 수 있습니다. [각 서비스는 비즈니스 기능을 제공하며, 플랫폼과 언어를 넘나들며 서로 통신할 수 있습니다](https://aws.amazon.com/ko/what-is/service-oriented-architecture/)[1](https://aws.amazon.com/ko/what-is/service-oriented-architecture/)[2](https://www.redhat.com/ko/topics/cloud-native-apps/what-is-service-oriented-architecture).

SOA의 주요 이점은 다음과 같습니다:

1. **출시 기간 단축**: 서비스를 재사용하여 빠르게 애플리케이션을 어셈블할 수 있습니다.
2. **효율적인 유지 보수**: 작은 서비스를 수정해도 전체 기능에 영향을 미치지 않습니다.
3. **더 뛰어난 적응성**: 기술 발전에 더 잘 적응하고 현대화할 수 있습니다.

SOA의 기본 원칙은 다음과 같습니다:

- **상호 운용성**: 서비스 간에 플랫폼이나 프로그래밍 언어에 관계없이 상호작용할 수 있어야 합니다.
- **느슨한 결합**: 서비스는 느슨하게 결합되어야 하며, 상태 비저장 서비스여야 합니다.
- **추상화**: 클라이언트는 서비스의 코드 논리를 알 필요 없이 서비스를 사용할 수 있어야 합니다.
- [**세분화**: 각 서비스는 하나의 개별 비즈니스 기능을 수행할 수 있도록 적절한 크기와 범위를 가져야 합니다](https://aws.amazon.com/ko/what-is/service-oriented-architecture/)[1](https://aws.amazon.com/ko/what-is/service-oriented-architecture/).

예를 들어, 조직 내의 여러 비즈니스 프로세스에서 사용자 인증 기능이 필요할 경우, 단일 인증 서비스를 생성하여 모든 애플리케이션에 재사용할 수 있습니다. [또한 의료 기관 내의 환자 관리 시스템과 전자 건강 기록 (EHR) 시스템에서 환자를 등록해야 하는 경우 해당 시스템들이 하나의 공통 서비스를 호출하여 환자 등록 태스크를 수행할 수 있습니다](https://aws.amazon.com/ko/what-is/service-oriented-architecture/)[1](https://aws.amazon.com/ko/what-is/service-oriented-architecture/).

<br>

**웹 서비스**는 네트워크 상에서 서로 다른 종류의 컴퓨터들 간에 상호작용을 하기 위한 소프트웨어 시스템입니다. [이는 **서비스 지향적 분산 컴퓨팅 기술**의 일종으로, 주요 표준 기술로는 **SOAP**, **WSDL**, **UDDI** 등이 사용됩니다](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4)[1](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4).

<br>

웹 서비스는 다음과 같은 특징을 가지고 있습니다:

<br>

- **느슨한 연결 (loose coupling):** 웹 서비스는 서로 독립적으로 작동하며, 강한 의존성 없이 상호작용합니다.

<br>

<br>

- **XML 유니코드 부호화:** 메시징에 XML을 사용하여 상호운용성을 높입니다.

- **메시지 지향 (message-oriented):** 객체 지향이 아닌 메시지 지향적인 특성을 가집니다.

<br>

웹 서비스는 **월드 와이드 웹**과 혼동해서는 안 됩니다. [월드 와이드 웹은 사람과 컴퓨터 간의 상호작용을 위한 시스템이며, 반면 웹 서비스는 컴퓨터와 컴퓨터 간의 상호작용을 위한 시스템입니다](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4)[1](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4)[2](https://developer.mozilla.org/ko/docs/Learn/Common_questions/Web_mechanics/Pages_sites_servers_and_search_engines).

<br>

웹 서비스는 \*\*서비스 지향 아키텍처 (SOA)\*\*의 기반이 되며, 최근에는 **REST 스타일 웹 서비스**도 주목받고 있습니다. [웹 서비스를 활용하여 여러 웹 서비스를 조합하여 웹 애플리케이션을 구축하는 것을 **매시업**이라고 부르며, 이 분야는 새로운 사업으로까지 확장되고 있습니다](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4)[1](https://ko.wikipedia.org/wiki/%EC%9B%B9_%EC%84%9C%EB%B9%84%EC%8A%A4)[2](https://developer.mozilla.org/ko/docs/Learn/Common_questions/Web_mechanics/Pages_sites_servers_and_search_engines).
